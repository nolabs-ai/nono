<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `nono` CLI ↔ test harness | `--dangerous-force-wfp-ready` is an internal test-only flag that forces WFP readiness without a real WFP service. Promoting it out of `cfg(debug_assertions)` exposes it to release-mode builds; the `hide = true` attribute keeps it absent from user-facing `--help` but a knowledgeable user could discover and invoke it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-04-01 | E (Elevation of Privilege) | If `--dangerous-force-wfp-ready` is exposed in release builds, a malicious or curious user could invoke it to force WFP to claim readiness without actual service backing, potentially bypassing block-net enforcement. | mitigate | Keep `hide = true` attribute so the flag is absent from `--help`. ALSO: the runtime setter at `exec_strategy_windows/mod.rs:397-402` must remain gated by additional preconditions (test-built binary identifier, environment variable, etc.) — Task 2 verifies whatever guard the setter uses is preserved when promoting the flag. If the only guard was `cfg(debug_assertions)`, add a stronger runtime guard before promoting. LOW severity in practice because the flag forces WFP into "ready" state without actually creating a privileged network filter; it neither escalates filesystem nor process privilege. |
| T-41-04-02 | T (Tampering) | Promoting the flag could enable test-environment WFP-bypass invocations in production deployments. | mitigate | The `dangerous-` prefix + `hide = true` + (verified) runtime guard limit the risk. Document the change in the commit body and in `crates/nono-cli/src/cli.rs` doc-comment above the flag. |
</threat_model>
