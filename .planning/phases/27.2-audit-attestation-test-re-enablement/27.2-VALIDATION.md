## Adversarial Stance Notes

The two BL-01/BL-02 tests are specifically positioned to catch silent reverts:

**BL-01 sentinel** — `audit_session_dir.join("session.json").exists()` is asserted BEFORE the `audit verify` exit-code check. Pre-fix, the audit dir holds only `audit-attestation.bundle` and `audit-events.ndjson` for combo sessions; session.json lives only in the rollback dir. The mirror block at `rollback_runtime.rs:658-671` is the SOLE writer of `<audit_root>/<id>/session.json` for combo sessions, so its presence is a structural signal that the mirror block ran. Removing the mirror block fails the assertion at the first hurdle (no need to wait for the verify subprocess to fail).

**BL-02 sentinel** — `nono rollback list --all --json` calling `discover_sessions` (which uses `rollback_session::rollback_root()`) MUST find the session. Under NONO_TEST_HOME, both helpers resolve identically so a path-component-revert isn't directly caught by path comparison; instead the test catches behavioral mismatches: any future refactor where `create_audit_state`'s rollback dir derivation drifts from `rollback_session::rollback_root()` will produce a session in a directory `discover_sessions` does not scan, and the `session_ids.contains(&combo_id)` assertion fires red. On Windows production hosts (no NONO_TEST_HOME), the divergence is direct (`%USERPROFILE%` vs `%LOCALAPPDATA%`) and the test catches it immediately.

Both tests were authored with the FORCE stance: starting hypothesis was that the BL-01/BL-02 fixes might be incomplete or might regress under iteration. After 1 debug iteration (the initial run failed because `--allow-cwd` was missing — Windows execution-directory allowlist guard rejected the workspace as a launch dir), both tests passed with the recipe documented in 27.2-REVIEW.md §BL-01.
