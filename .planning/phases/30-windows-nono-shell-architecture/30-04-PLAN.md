```markdown

### Security envelope under `nono shell` on Windows (Phase 30)

The sandboxed shell child runs under a **Low Integrity primary token** (Phase 30 D-01). Filesystem write enforcement uses Windows Mandatory Integrity Control (MIC) via per-path NO_WRITE_UP labels:

- **Writes to paths OUTSIDE the grant set** are denied at OS level — kernel `MandatoryIntegrityCheck` runs before DACL evaluation. Unlabeled files (the default for `~/Desktop`, `C:\ProgramData`, etc.) are treated as Medium-IL; a Low-IL subject cannot write to them. `Out-File` returns `Access to the path '...' is denied.`
- **Writes to paths INSIDE the grant set** (e.g. `~/.claude` if `--profile claude-code` includes it) are bounded by the `CapabilitySet` contract, NOT by MIC. The Low-IL subject and the Low-IL labeled object share the same integrity level, so MIC permits the operation; the capability check at the application layer is what enforces read-only vs read-write semantics for grant-set paths.
- **Reads of granted paths** use the same mandatory-label mechanism — Low-IL subjects can read Medium-IL objects (TokenDominates=FALSE in the read direction is permissive by default unless NO_READ_UP is explicitly set; Write-only grants get NO_READ_UP applied; ReadWrite/Read grants do not).

Per-session WFP differentiation via the synthetic restricting SID is **not used on this path**; outbound network filtering falls back to AppID-based filtering (same security-envelope shift Phase 15 documented for the `nono run --detached` path on Windows). Network enforcement still requires the `nono-wfp-service` to be running on the test machine — without it, network filtering reverts to no-op (a separate v2.4 follow-up; the per-user MSI does not ship the WFP service today).

The Claude Code PreToolUse hook is **defense-in-depth** on top of the OS-level write-deny — it surfaces user-facing messages but is not the primary boundary. If the hook does not fire (separate concern tracked in `.planning/debug/claude-code-hook-not-firing.md`), the OS-level write-deny still applies.

For the technical background (token shape selection, MIC pre-DACL pseudocode, capability-pipe SDDL admit, Phase 15 parallel finding), see `.planning/phases/30-windows-nono-shell-architecture/30-RESEARCH.md`.
```
