title: Windows POC Handoff Cookbook
description: Step-by-step recipe for testing the current Windows nono binary with Claude Code before shipping to a small POC group
---

Use this cookbook to validate a Windows nono build with Claude Code on your own machine, then package it for a 3–5 user POC group. It is the operator-side complement to [Windows Preview Pilot](/cli/development/windows-preview-pilot) — that doc describes the contract; this one walks you through the build, install, validate, distribute loop end-to-end.

Scope: the **current** Windows supported command surface (`run`, `shell`, `wrap` via ConPTY, file/network grants enforced by Job Objects + per-path mandatory integrity labels). For the broader contract and what is intentionally unsupported, see [Windows Filesystem Parity Contract](/cli/development/windows-filesystem-parity-contract) and [Windows Feature Gap Matrix](/cli/development/windows-feature-gap-matrix).

<Note>
`nono run -- <command>` is the supported invocation for non-interactive POC commands on Windows. As of Phase 31 (validated 2026-05-09), `nono shell --profile claude-code --allow-cwd` is also supported as the recommended Windows TUI host for `claude` and similar interactive agents — see [Windows `nono shell` — security envelope](#windows-nono-shell--security-envelope-phase-31-validated-2026-05-09) for the broker-process architecture and OS-level enforcement story.
</Note>

## Prerequisites

- Windows 10 build 17763+ or Windows 11 on the **test** machine (clean VM strongly preferred for the dress rehearsal).
- Rust toolchain (matches `rust-toolchain.toml` in this repo) on the **build** machine.
- For MSI packaging: WiX 7 or later (`dotnet tool install --global wix --version 7.0.0`). WiX 7 requires the .NET 8 SDK, which the GitHub `windows-latest` runner already ships. The build script passes `-acceptEula wix7` to acknowledge the [FireGiant OSMF](https://docs.firegiant.com/wix/osmf/) EULA non-interactively (required by v7 enforcement of `WIX7015`; nono is below the $10K/yr fee threshold).
- A Claude Code installation on the test machine (`claude --version` works in PowerShell).

## Step 1 — Pick or produce a binary

Three paths, in order of formality.

### Option A: Use a binary already in this tree

Fast, dev-box only. Do **not** ship a `target/`-built binary to POC users — give them a versioned MSI or zip.

```powershell
.\target\release\nono.exe --version
# nono 0.37.1
```

### Option B: Fresh release build on this machine

```powershell
make build-release-cli
# or, equivalently:
cargo build -p nono-cli --release
```

Result: `target\release\nono.exe`.

### Option C: Versioned MSI for POC users

Per-user is the right choice for a POC — no admin install, no machine-wide footprint, no service registration to undo.

```powershell
# Build BOTH binaries for the explicit Windows MSVC target. nono-shell-broker.exe
# is required by the MSI as of Phase 31 Plan 04 — it installs as a sibling of
# nono.exe so the runtime cascade in spawn_windows_child can resolve it via
# current_exe().parent() (D-07).
cargo build --release --target x86_64-pc-windows-msvc -p nono-cli -p nono-shell-broker

# Per-user MSI
.\scripts\build-windows-msi.ps1 `
  -VersionTag v0.37.1-poc.1 `
  -BinaryPath .\target\x86_64-pc-windows-msvc\release\nono.exe `
  -BrokerPath .\target\x86_64-pc-windows-msvc\release\nono-shell-broker.exe `
  -Scope user `
  -OutputDir dist\windows
```

Output: `dist\windows\nono-v0.37.1-poc.1-x86_64-pc-windows-msvc-user.msi`.

<Warning>
The **machine** MSI (`-Scope machine`) registers `nono-wfp-service` only when you explicitly pass `-ServiceBinaryPath .\target\x86_64-pc-windows-msvc\release\nono-wfp-service.exe`. The release build DOES produce that binary (`make build-release-cli` → `cargo build --release -p nono-cli`), but the MSI script defaults `-ServiceBinaryPath` to empty — so a machine MSI built without that flag installs successfully but never registers the WFP service. Stay on `-Scope user` for the POC; per-user is the right choice anyway (no admin install, no service to undo).
</Warning>

If you want a portable archive instead of MSI:

```powershell
Compress-Archive `
  -Path .\target\x86_64-pc-windows-msvc\release\nono.exe, `
        .\target\x86_64-pc-windows-msvc\release\nono-shell-broker.exe `
  -DestinationPath .\dist\nono-v0.37.1-poc.1-x86_64-pc-windows-msvc.zip
```

POC users extract to a stable directory (e.g. `C:\Tools\nono`) and add it to PATH.

## Step 2 — Stage on the test machine

For the **dev-box** validation pass, drop the binary on PATH for one shell:

```powershell
$env:Path = "$PWD\target\x86_64-pc-windows-msvc\release;$env:Path"
nono --version
```

For a **clean test machine** (recommended before POC handoff — your dev box has muddy state from prior runs), install the way a POC user would:

```powershell
# Per-user MSI install. No elevation prompt.
msiexec /i .\nono-v0.37.1-poc.1-x86_64-pc-windows-msvc-user.msi
# Open a fresh PowerShell so PATH picks up the new entry, then:
nono --version
```

## Step 3 — Pre-flight checks

```powershell
nono setup --check-only
```

What you should see (paraphrased from a real run on a Standard User session):

- `nono binary found at <path>`, `Version: 0.37.x`, `Platform: Windows (restricted execution)`
- `Support status: supported`
- `Job Object creation: OK`, `Token Integrity level support: OK`, `BFE service status: OK`
- `WFP readiness: missing binary` — **expected** for the POC. The `claude-code` profile does not require WFP; this only matters for domain-level network filtering, which is out of scope for the POC.
- `35 sensitive paths blocked by default`, `26 dangerous commands blocked by default`
- `claude-code` listed under built-in profiles.

### One-time fix: `.claude.json` symlink warning

The first time Claude Code is run under `nono` on a fresh user profile, you will see:

```text
WARN Failed to create ~/.claude.json symlink on Windows
     (C:\Users\<u>\.claude.json -> .claude\claude.json):
     A required privilege is not held by the client. (os error 1314);
     claude-code token refresh may require manual setup or Developer Mode
```

This is **fail-open** — Claude still works, but token refresh writes land outside the sandbox-allowed `~/.claude\` tree. Pick one fix:

1. **Enable Developer Mode** (preferred for POC users, no admin needed): `Settings → System → For developers → Developer Mode = On`. Symlink creation then succeeds for that user.
2. **Run the first hook install elevated** (admin shell, just once): run `nono setup` (without `--check-only`) once as admin to create the symlink. Subsequent runs are unprivileged.

Communicate one of these in your POC handoff. Do not let users discover the warning themselves.

## Step 4 — First Claude Code run (installs the hook)

```powershell
# Dry-run first — verify the policy is what you expect
nono run --dry-run --profile claude-code -- claude --version
```

Expected output shape:

```text
nono v0.37.x
  Updating claude-code hook (new version available)
  Capabilities:
   r+w  \\?\C:\Users\<u>\.claude (dir)
   r+w  \\?\C:\Users\<u>\.cache\claude (dir)
       + 6 system/group paths (-v to show)
   net  outbound allowed
  dry-run sandbox would be applied with above capabilities
  $ claude --version
```

The `Updating claude-code hook` line indicates that `~/.claude/hooks/nono-hook.sh` and the upfront sandbox section in `~/.claude/CLAUDE.md` were written. The install is idempotent and is skipped on subsequent runs unless the hook version changed.

For the POC, use `nono run -- <command>` for non-TUI commands. TUI agents (claude, codex, etc.) host inside `nono shell --profile claude-code --allow-cwd` via the broker-process architecture — see [Windows `nono shell` — security envelope](#windows-nono-shell--security-envelope-phase-31-validated-2026-05-09) below.

## Sigstore Trust Root Setup (one-time per user)

Keyless `nono trust verify` requires a refreshed Sigstore trusted root cached
at `~/.nono/trust-root/trusted_root.json`. The cache is per-user (no admin)
and refreshes only when you explicitly run setup — verify NEVER attempts
inline network refresh. Mirror of the WFP service prereq pattern from Step 5
(below): explicit operator action up front, fail-closed-with-recovery-hint at
runtime, no implicit network on the verify path.

Run once after install:

```powershell
nono setup --refresh-trust-root
```

The command fetches TUF metadata from `https://tuf-repo-cdn.sigstore.dev` and
writes the verified trusted root to your per-user cache. Subsequent keyless
`nono trust verify` invocations run offline against that cache.

**Fail-closed contract:** if the cache is absent or expired, keyless
`nono trust verify` fails with a recovery message naming the exact command
(`nono setup --refresh-trust-root`) — see [Sigstore TUF Cache ADR](../../architecture/sigstore-tuf-cache.md)
for the design rationale (Phase 32 D-32-01). Verify-is-offline is structural
and dynamic — there is no inline network call on the verify path at all.

#### Known issue: Sigstore TUF root rotation (sigstore-verify 0.6.5)

If `nono setup --refresh-trust-root` fails with:

```
ERROR Setup error: Failed to fetch Sigstore trusted root from
  https://tuf-repo-cdn.sigstore.dev: TUF error: TUF repository load failed:
  Failed to verify trusted root metadata:
  Signature threshold of 3 not met for role root (0 valid signatures)
```

…then your binary is hitting a known upstream issue: `sigstore-verify 0.6.5`
(the fork's pinned dependency) embeds a TUF trust anchor that Sigstore has
since rotated out. The TUF fetch succeeds; signature validation fails because
zero of the embedded anchor keys are present in the current published root.

The verify path is structurally offline (Phase 32 D-32-15), so a manually-placed
`trusted_root.json` works around this. The repo ships a maintainer-captured
known-good root at `crates/nono/tests/fixtures/trust-root-frozen.json`
(captured 2026-05-10 from `sigstore/root-signing@main`). Drop it into your
cache directly:

```powershell
# Workaround until the fork upgrades to sigstore-verify 0.6.6+ (tracked as
# P32-DEFER-005 in .planning/phases/32-sigstore-integration/deferred-items.md).
$cacheDir = "$env:USERPROFILE\.nono\trust-root"
New-Item -ItemType Directory -Force -Path $cacheDir | Out-Null
Invoke-WebRequest -UseBasicParsing `
  -Uri "https://raw.githubusercontent.com/oscarmackjr-twg/nono/281f71ab/crates/nono/tests/fixtures/trust-root-frozen.json" `
  -OutFile "$cacheDir\trusted_root.json"
```

After that, `nono trust verify` with `--issuer` + `--identity` works offline
against the cached file. `nono setup --check-only` should report
`Trust root cache: OK` once the file is in place.

The `--refresh-trust-root` path will start working again once the fork's
`sigstore-verify` dep is upgraded to 0.6.6+ (which embeds the rotated TUF
anchor); until then, the manual file drop is the supported workaround for
POC use.

**Check status:**

```powershell
nono setup --check-only
```

Look for the `Trust root cache:` line — `OK` (with cache path), `STALE` (with
diagnostic), or `NOT INITIALIZED` (run `--refresh-trust-root` to fix). The same
output also includes `self-authenticode-subject:` and `self-authenticode-thumbprint:`
lines that report `nono.exe`'s own signature — what the broker self-trust-anchor
gate compares against (see Broker.exe Verification subsection below).

### Verifying Keyless-Signed Artifacts

Keyless `nono trust verify` requires explicit `--issuer` (OIDC issuer URL,
exact match) and `--identity` (regex against the bundle SAN). Both flags are
fail-closed-mandatory — verify rejects bundles when either is missing.
Providing both is what selects keyless mode; there is no separate `--keyless`
boolean on verify.

**GitHub Actions release** (canonical pattern):

```powershell
nono trust verify `
  --issuer https://token.actions.githubusercontent.com `
  --identity '^https://github\.com/<org>/<repo>/\.github/workflows/release\.yml@refs/tags/v.*$' `
  <bundle-file>
```

Example for the `nono` project itself:

```powershell
nono trust verify `
  --issuer https://token.actions.githubusercontent.com `
  --identity '^https://github\.com/always-further/nono/\.github/workflows/release\.yml@refs/tags/v.*$' `
  instruction.md
```

**GitLab CI release**:

```powershell
nono trust verify `
  --issuer <gitlab-issuer-url> `
  --identity '<gitlab-identity-pattern>' `
  <bundle-file>
```

A baked-in policy template lives at
[`docs/templates/trust-policy-keyless-template.json`](../../templates/trust-policy-keyless-template.json)
— copy to `~/.nono/trust-policy.json` to make `--policy` invocations match the
project's release pipeline by default.

**Local development:** keyless signing is CI-only (no interactive browser OAuth).
For local dev, use:

```powershell
nono trust sign --keyref <key-path> <file>
nono trust verify <file>  # keyed verify; no --issuer/--identity needed
```

### Broker.exe Verification at Launch

`nono shell --profile <name>` on Windows spawns `nono-shell-broker.exe` (a sibling
of `nono.exe` in the install directory). Phase 32 adds an Authenticode
self-trust-anchor verification before the spawn: `nono.exe` extracts ITS OWN
signature, then requires `broker.exe`'s signature to match (subject + thumbprint).

On a properly-installed release MSI, both binaries are signed by the same
pipeline identity — verification passes silently. The `nono setup --check-only`
diagnostic surfaces the exact subject + thumbprint nono.exe is comparing against.

**Fail mode:** if a malicious sibling broker is dropped in, or if either
binary's signature is unverifiable, `nono shell` fails with text containing:

```
Trust verification failed: broker.exe Authenticode signature does not match nono.exe
```

— the message names both expected and observed `(subject, thumbprint)` pairs.
No escape-hatch flag exists by design. Recovery: reinstall from the official MSI.
See [Broker Trust Anchor ADR](../../architecture/broker-trust-anchor.md) for the
threat model and self-bootstrapping rationale (Phase 32 D-32-13/14).

**Dev-build note:** `cargo build`-built `nono.exe` runs out of `target/(debug|release)/`
— the install-layout substring detector activates and SKIPS verification (logged at
info-level under `tracing` target `broker_authenticode`). Production installs
(`Program Files\nono\` or `LocalAppData\Programs\nono\`) do not match this pattern
and always enforce verification. The detector is a runtime check on `current_exe()`'s
path, NOT `#[cfg(debug_assertions)]` — `cargo test --release` therefore exercises
the production-mode strict path correctly (Phase 32 Pitfall 6).

## Step 5 — Smoke checklist

Run these from a fresh PowerShell. Capture stdout, stderr, and `nono --version` for each — that is the minimum triage payload from [Windows Preview Pilot § Debugging Inputs](/cli/development/windows-preview-pilot#debugging-inputs).

### Must-pass

```powershell
# Identity & support
nono --version
nono --help | Select-String "shell|wrap|run"
nono setup --check-only

# Dry-run shapes (verify capability lists print as expected)
nono run --dry-run --profile claude-code -- claude --version
nono run --dry-run --allow . --read $env:USERPROFILE\.claude -- claude --version

# Live, narrowest grant — works without profile (non-interactive child, pipes are fine)
nono run --allow . -- cmd /c echo hello

# Live with profile, non-interactive (exercises policy + spawn through anonymous pipes)
nono run --profile claude-code -- cmd /c echo "claude-code profile spawned ok"

# Exact-file grant (Windows supports this)
$tmp = New-TemporaryFile
"hello" | Out-File $tmp.FullName
nono run --read-file $tmp.FullName -- cmd /c type $tmp.FullName

# Block-net requires the WFP service to be registered + running first. Run this ONCE,
# in an elevated PowerShell (admin), before any block-net or network-profile smoke:
#   nono setup --install-wfp-service
#   nono setup --start-wfp-service
# (Skip if `nono setup --check-only` already shows the service is registered & running.)
# Use a non-interactive child so the smoke is a one-liner. (Interactive `nono shell` is
# also supported as of Phase 31 — see the security-envelope section for the broker path.)
nono run --profile claude-code --block-net -- cmd /c "echo block-net ok && curl -m 3 https://example.com"
# The curl call should fail/timeout since outbound is blocked. Verify exit and no body returned.
# If you instead see the diagnostic `This request remains fail-closed until WFP activation
# is implemented`, the WFP service prereq above was skipped — fix that and re-run. (This
# is fail-closed-correct behavior, NOT a regression.)

# Why-query: structured policy introspection
nono why --path $env:USERPROFILE\.ssh\id_rsa --op read
nono why --json --path $env:USERPROFILE\.aws --op read
```

### Must-fail loudly

These flows depend on the WFP backend. They must surface a backend-readiness diagnostic — not silently allow the request — when the WFP service is not installed:

```powershell
# Domain-level filtering (proxy/WFP)
nono run --profile claude-code --network-profile claude-code -- claude --version

# Domain allowlist add
nono run --profile claude-code --allow-domain api.openai.com -- claude --version
```

Expected stderr (paraphrase — exact text varies by target/path/backend summary):

```text
Windows WFP runtime activation is required for <target> but the WFP service binary
`<path>` is missing from this build output. Run `cargo build -p nono-cli --bins` first
(<backend_summary>). This request remains fail-closed until WFP activation is implemented.
```

Grep operators can anchor on: `This request remains fail-closed until WFP activation is implemented`. If you see exit code 0 and no stderr line containing that anchor, the fail-closed gate has regressed — stop the POC.

If either of those silently runs, **stop the POC**. That is a fail-closed regression and worth a bug report before any user sees it.

## Known limitation: `nono run` cannot host TUI agents on Windows

`nono run` on Windows always opens anonymous pipes for the child process — never a ConPTY. Non-interactive children (`cmd /c echo`, `cargo build`, `python -c '...'`, `claude --version`) work fine through pipes. Full-screen TUI agents (`claude`, `codex`, anything that needs cursor positioning, alternate screen buffer, or interactive stdin) cannot render through `nono run` and will appear to hang.

The constraint is documented in `crates/nono-cli/src/supervised_runtime.rs:101-111`: combining `PROC_THREAD_ATTRIBUTE_PSEUDOCONSOLE` with `DETACHED_PROCESS` historically crashed grandchildren with `STATUS_DLL_INIT_FAILED (0xC0000142)` (see `.planning/debug/resolved/windows-supervised-exec-cascade.md`). The conservative Phase-15 fix disabled PTY allocation on the **detached** Windows supervised path. Interactive `nono shell` does allocate ConPTY post-Phase 31 via the broker process — see security envelope below.

**For the POC: use `nono run -- <non-TUI command>` for any non-interactive child on Windows. For TUI agents (claude, codex, etc.), use `nono shell --profile claude-code --allow-cwd`** — see [Windows `nono shell` — security envelope](#windows-nono-shell--security-envelope-phase-31-validated-2026-05-09) below for the broker-process architecture that delivers full TUI rendering with OS-level mandatory-label write enforcement intact.

## Windows `nono shell` — security envelope (Phase 31, validated 2026-05-09)

`nono shell --profile claude-code --allow-cwd` on Windows 10/11 is the recommended interactive TUI host for `claude` and similar agents. The supported token shape ships via the broker-process architecture (Phase 31, validated on the user's Windows test box on 2026-05-09 after a same-day PoC validation 2026-05-08). It addresses the Phase 30 final-state finding that no **direct** Low-IL primary-token spawn from `nono.exe` can host a ConPTY child without tripping `STATUS_DLL_INIT_FAILED (0xC0000142)` at the CSRSS console-subsystem ALPC handshake during `KernelBase.dll` DllMain.

> **Freshness note:** Phase 31 production validation completed on 2026-05-09 (the day this cookbook was written). The broker path has minimal soak time on `main`. Day-1 POC users should report any anomalies promptly via the [Windows Preview Pilot § Debugging Inputs](/cli/development/windows-preview-pilot#debugging-inputs) triage payload, and operators should consider deferring the actual handoff by 24-48h to allow follow-up patches to land.

### Token shape

- **`nono.exe`** runs at the user's identity (Medium IL or higher). It is the supervisor and policy enforcement entry point.
- **`nono-shell-broker.exe`** is a sibling binary of `nono.exe`, Authenticode-signed with the same key (Phase 31 release pipeline). `nono.exe` spawns the broker at the user's identity (Medium IL) via `CreateProcessW(EXTENDED_STARTUPINFO_PRESENT)` with a `PROC_THREAD_ATTRIBUTE_HANDLE_LIST` that scopes inherited handles to the broker. The broker exists *because* directly spawning a Low-IL child from `nono.exe` triggers `STATUS_DLL_INIT_FAILED (0xC0000142)` at CSRSS attach time — the Low-IL ALPC client cannot reach the CSRSS console handler port, KernelBase's `ConClntInitialize` fails, the DllMain returns FALSE, and the loader exits the process before reaching `main()`.
- **Token degradation inside the broker.** `nono-shell-broker.exe` calls `DuplicateTokenEx(SecurityAnonymous, TokenPrimary)` against its own token, then `SetTokenInformation(TokenIntegrityLevel, WinLowLabelSid)` to lower the duplicate token to the **Low Mandatory Integrity Level**. The original Medium-IL token is preserved; only the duplicate is dropped.
- **The actual shell child** (PowerShell 5.1 or `cmd.exe`) is launched by the broker via `CreateProcessAsUserW(low_il_token, ..., dwCreationFlags=EXTENDED_STARTUPINFO_PRESENT)`. **NO `CREATE_NEW_CONSOLE`. NO `PROC_THREAD_ATTRIBUTE_PSEUDOCONSOLE`.** The Low-IL child INHERITS the broker's already-attached console; KernelBase short-circuits the CSRSS attach because a console handle is already inherited (RESEARCH Assumption A1, validated end-to-end on the production binary chain).
- **The Low-IL shell child is the user-facing shell**. Mandatory-label `NO_WRITE_UP` enforces write-deny via the MIC kernel pre-DACL check: a Low-IL subject (the shell process) cannot write to a Medium-IL object (paths outside the grant set, e.g., the user's Desktop, `AppData\Roaming` outside the granted subtree), regardless of the file's DACL.

### What's enforced at OS level

- **Filesystem write-deny outside the grant set.** Mandatory-label `NO_WRITE_UP` at the MIC kernel pre-DACL check. This is kernel-level enforcement — NOT a hook check, NOT a userland policy. A `Set-Content -Path 'C:\Users\<u>\Desktop\evil.txt' -Value 'x'` attempt from inside the sandboxed shell raises `UnauthorizedAccessException` and the file does not exist on disk.
- **Reads of granted paths still succeed.** Per-path Low-IL labels with read-pass mode masks via `try_set_mandatory_label` (Phase 21 carry-forward). `Get-Content $env:USERPROFILE\.claude\claude.json` works inside the sandboxed shell when the profile grants read access to that path.
- **Job Object containment.** The broker's process tree is capped by a Job Object created by `nono.exe`. The Low-IL grandchild inherits Job Object membership automatically because `JOB_OBJECT_LIMIT_*BREAKAWAY*` flags are unset (D-04 acceptance — `cargo test broker_dispatch_tests` 2/2 PASS includes an `IsProcessInJob` runtime assertion). RESL caps (`--memory`, `--cpu-percent`, `--max-processes`, `--timeout`) cascade from the Job Object.

### What's defense-in-depth (NOT the primary boundary)

- **Claude Code `PreToolUse` hook.** The hook intercepts attempted writes to disallowed paths from inside the agent and surfaces a prompt. If the hook fails (or is misconfigured, or is bypassed by an attacker controlling the agent's actions), the OS-level mandatory-label check still denies the write. The OS check is the security boundary; the hook is UX + audit. Treat the hook as belt; the mandatory label is suspenders.
- **Network differentiation per-session via WFP `FWPM_CONDITION_ALE_USER_ID` is waived on this path** (broker→Low-IL child); falls back to AppID-based filtering (same as Phase 15 detached-path waiver). For full per-session network filtering (granular `--allow-domain` differentiation across concurrent sessions of the same binary), use `nono run` (non-PTY supervised path retains `WRITE_RESTRICTED` + per-session SID).

### What's NOT enforced

- **Read-deny across the IL boundary is not enforced by default.** Low-IL does not block reads of Medium-IL files (NO_READ_UP is NOT set on the kernel default mandatory label). For read-deny on specific paths, use single-path grants with explicit Low-IL labels via `nono shell --read-only-grants` (Phase 21 design, carry-forward).
- **Network deny-within-allow.** The Windows WFP allow-list is positive-only (no Landlock-style deny-within-allow). Use a profile-scoped `--allow-domain` allowlist for the supported invocation, not exclusion lists.

### Audit trail

The Authenticode chain-walker (`AUDC-01..03`, Phase 28) records `signer_subject` + SHA-1 thumbprint for both `nono.exe` AND the broker process — both signed with the same key, so the audit trail records a consistent identity for the supervisor stack across the launch chain.

Operators can verify post-run with `nono audit list --recent 5` (or `nono audit list --json` for structured output, plus `nono audit show <session-id> --json` for per-session detail) to see the recorded `signer_subject` and SHA-1 thumbprint per launch. If the audit trail is empty after a `nono shell` invocation, the AUDC chain-walker is misconfigured — out of scope for the POC smoke but worth flagging in the triage payload.

```powershell
.\nono.exe shell --profile claude-code --allow-cwd
# Inside the sandboxed shell:
claude  # full TUI rendering — alternate screen buffer, cursor positioning, raw-mode input
```

For non-TUI commands, `nono run --profile claude-code -- <command>` remains the supported invocation (anonymous-pipe stdio; `WRITE_RESTRICTED` + per-session SID; full WFP per-session differentiation).

## Step 6 — POC user handoff content

Drop the table below into the README you ship alongside the MSI. These are the items POC users will hit and need to know about.

| Item | What to tell users |
|------|---|
| Install | `msiexec /i nono-vX-x86_64-pc-windows-msvc-user.msi`, then open a fresh PowerShell |
| First-time symlink warning | Enable Developer Mode (`Settings → System → For developers`) **or** run `nono setup` once as admin to create the symlink. Benign if skipped, but token refresh writes leak outside `~/.claude\`. |
| Non-TUI commands (cargo, npm, scripts) | `nono run --profile claude-code -- <command>` is the supported invocation for non-interactive children |
| Quick verification | `nono run --profile claude-code -- claude --version` exercises the full sandbox + profile path without needing a TUI |
| TUI agents (claude, codex, etc.) | `nono shell --profile claude-code --allow-cwd`, then run `claude` (or other TUI) inside. The broker-process architecture (Phase 31, validated 2026-05-09) delivers full TUI rendering with OS-level mandatory-label `NO_WRITE_UP` write-deny intact. See [Windows `nono shell` — security envelope](#windows-nono-shell--security-envelope-phase-31-validated-2026-05-09) for the security model and what's enforced vs. defense-in-depth. |
| Triage payload on a bug | Exact command line, full stdout, full stderr, `nono --version`, `claude --version`, `nono setup --check-only` output, Windows build (`(Get-CimInstance Win32_OperatingSystem).BuildNumber` or `[Environment]::OSVersion.Version.Build`), whether WFP was reported missing. |
| Not yet supported on this build | Domain-level network filtering (`--network-profile`, `--allow-domain`) — needs WFP service, which the per-user MSI does not ship today. Use `--block-net` for offline or unrestricted network for online. (`nono shell` per-session WFP differentiation is waived on the broker path; falls back to AppID-based filtering — see security envelope above.) |
| Uninstall | Apps & Features → "nono" → Uninstall (MSI). For zip: delete the extracted directory and remove from PATH. |

## Step 7 — Loop yourself before shipping

A reasonable validation order on your build box:

1. `make build-release-cli` (or use an existing `target\release\nono.exe` if it is current).
2. Build the per-user MSI per Step 1 Option C with a real version tag (e.g. `v0.37.1-poc.1`).
3. Install the MSI on this machine **or a separate VM**. A clean VM is the only way to catch missing-PATH and missing-Developer-Mode footguns that your dev box hides.
4. Run the full Step 5 checklist from a **fresh** PowerShell window.
5. If any negative test in Step 5 silently passes → stop, fix, rebuild. Do not ship.
6. Hand the MSI plus the Step 6 table to the first POC user and watch their first `setup --check-only` over their shoulder. The first 10 minutes of the first POC user's session will surface anything the dress rehearsal missed.
